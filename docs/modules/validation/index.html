Validation

Validation#

Effective business intelligence efforts starts with high quality data -- Simran Jindal, IBM

Overview#

Data quality is an important aspect for any business process. Having correct data is essential to make the appropriate decisions and as such, a high data quality is one of the top assets a company can have.

To ensure that the correct data ends up in a system, the quality of the data needs to be as high as possible before loading the data into any of the target systems. In other words, the data needs to be verified at the gate.

Schematix Validation provides a wide range of possibilities to support such scenarios. Using a standard, centrally managed data schemas, validating data is a fast and easy process, no matter the amount of data being processed.

Concepts#

Schematix Validator can validate input data in numerous formats, such as pipe delimited formated, comma separated files (CSV), fixed width formats, JSON, XML and even binary data, all from a common, programming language agnostic. schema.

In order to validate data, the following concepts are defined within the validation process:

  • Input data
  • The validation schema
  • The mapping file (required for fixed width mapping only)
  • Validation reports

Basic usage#

The simplest way to validate an input file is as follows:

./schematix validate input.csv example.schema.json

This validates the input file (input.csv) using a validation schema (example.schema.json) which will contain the validation rules specific to this input file. Any records which do not adhere to the rules defined will be marked invalid and listed in the output report.

Validating CSV files#

To valid a CSV file with transactions for example, the following command can be used:

./schematix validate transactions.csv transactions.schema.json

This will read the transaction comma separate data using the specified schema. An output report will be displayed with the results of this validation.

Validating Fixed Width files#

To validate fixed width formats using a schema, an extra mapping file is needed to define the mapping between the schema definitions and the input format.

For example, if the input has a format like this:

2013-10-30 vQdpBmVCYMAr3ZWUST3lYxD 60.4075231

Then this could mean that the input contains 3 fields (for example, the transaction date, the reference and the amount). In the above input, the first 12 positions (10 plus 2 whitespace) are taken by the transaction date, followed by 25 characters for the reference, and finally 10 characters dfor the amount.

This leads to the following (simplified) mapping file:

{
"title": "Transaction mapping",
"type": "object",
"properties": {
"transaction_date": {
"width": 12
},
"reference": {
"width": 25
},
"amount": {
"width": 10
}
}
}

To use this file, the validator can be used in the following way:

./schematix validate --mapping=transactions.mapping.json transactions.csv transactions.schema.json

This will use the mapping file to validate the input, based on the supplied schema, and a report will be generated to output the results.

Reports#

Schematix supports multiple validation reports, depending on your requirements:

  • Console report - outputs to the console
  • CSV report - creates a CSV report with the incorrect records
  • JSON report - creates a JSON report with the incorrect records
  • HTML report - creates a simple HTML report

Console report#

By default, Schematix will show the validation output to the console, and will indicate which records are invalid and why.

CSV report#

When a more structured output is neded, the CSV report can be used. It lists only the failed records with the original structure, plus 1 additional column which contains the error message(s).

JSON report#

The JSON report is similar to the CSV report, but then in JSON format. It lists only the failed records with the original structure, plus 1 additional field which contains a list of error message(s).

Advanced usage#

Schematix Validation provides more ways to customise the validation process. This section will list most of them.

Skipping header and footers#

When a file contains a header and/or footer which shouldn't be validated, it's possible to skip validation on them. You can do so by specifying the number of lines to skip for either the header or footer, or both, like so:

./schematix validate --header-count=1 --footer-count=3 transactions.csv transactions.schema.json

In the above example, the first line and the last 3 lines will be excluded from validation.
